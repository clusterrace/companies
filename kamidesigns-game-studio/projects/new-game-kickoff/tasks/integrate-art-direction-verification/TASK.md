Wire `scripts/verify_art_direction.py` into the existing visual validation capture pipeline as an advisory post-step. Every visual validation run should now produce both a functional report and an art direction report.

## Context

Visual validation (capture screenshots via xvfb + engine CLI, upload for review) is a separate concern from art direction verification (compare captures against a style guide). The two should run in sequence: capture first, then verify. Both are advisory by default in this task — neither blocks CI.

## Protocol

1. Identify the current capture flow for this project. It likely involves SSHing to a rendering host, pulling the latest code, clearing engine caches, running the engine with `--write-movie` or equivalent, and SCPing frames back.
2. Add a post-step that invokes:
   ```
   python scripts/verify_art_direction.py \
       --captures <capture_dir> \
       --style-guide design/art/ \
       --vlm-model claude-haiku-4-5 \
       --vlm-samples 3
   ```
   (Without `--blocking`. Advisory only in this task.)
3. Commit the generated HTML + JSON reports to `reports/art_direction/<timestamp>.{html,json}` for human review.
4. For any `FAIL` verdict, spawn `qa-tester` to create one triage issue per unique failing rubric-item × capture pair, assigned to `art-director`. Each triage issue must include: capture filename, reference image(s) cited, rubric item ID, severity, and the VLM rationale verbatim.

## Configuration

- **Secrets**: none needed (uses OAuth via the local `claude` CLI)
- **Runtime**: Python 3.10+ with Pillow, numpy, PyYAML (see `scripts/requirements.txt` from the implementation task)
- **Budget**: approximately $0.04 per VLM sample with Haiku. For 7 scenes × 3 samples × (n_rubric_items fitting in one call) ≈ $1–2 per run. Track in the project budget.

## Hard rules

- **Do NOT enable `--blocking` mode.** Advisory only until per-rubric-item stability data exists.
- **Do NOT create triage issues for Layer 1 warnings** — only for Layer 1 blockers and Layer 2 `fail` verdicts.
- **Do NOT run verification without golden captures committed** — the `reference_similarity` check is toothless without them.
- **Do NOT attempt to auto-fix flagged issues.** Verification is read-only over captures and style guide; fixes go through normal issue/PR flow.

## Stability tracking

Write a `reports/art_direction/stability.json` that aggregates per-rubric-item pass/fail/insufficient_evidence rates across the last N runs. This is the dataset that eventually justifies promoting a rubric item from advisory → blocking (many runs later, when false-positive rate is genuinely low for that specific item). Do not promote anything in this task.

## Deliverables

- Updated capture pipeline (script, CI config, or wherever the current flow lives)
- First real end-to-end run, with the HTML report linked in a comment on this task
- Any triage issues generated by the first run, linked in a comment
